## High Findings

### H1. `CoopBoardNode` type incompatible with `@xyflow/react` `Node<T>` constraint [NEW]
- **File**: `packages/app/src/views/Board/index.tsx:63,86,265`
- **Issue**: `@xyflow/react`'s `Node<T>` requires `T extends Record<string, unknown>`, but `CoopBoardNode` is a named interface without an index signature. This produces 4 TS2344/TS2322 errors:
  - Line 63: `NodeProps<Node<FlowNodeData>>` -- constraint violation
  - Line 86: `Node<FlowNodeData>[]` in `mapNodes` -- same constraint
  - Line 265: Passing `board.nodes` to `<ReactFlow nodes={...}>` -- assignability error
- **Impact**: Type checking is broken for the entire Board view. Prop mismatches on `CoopBoardNodeCard` or `ReactFlow` would go undetected.
- **Confidence**: HIGH
- **Recommendation**: Add an index signature to `CoopBoardNode` in shared (`[key: string]: unknown`), or use a type assertion at the boundary where xyflow `Node` is constructed.

### H2. `newestCapture` type mismatch: `undefined` passed where `null` expected [NEW]
- **File**: `packages/app/src/app.tsx:620`
- **Issue**: `newestCapture` is derived as `captures[0]?.capture` (type `ReceiverCapture | undefined`), but `CaptureViewProps.newestCapture` expects `ReceiverCapture | null`. TypeScript TS2322 error.
- **Impact**: At runtime, both `undefined` and `null` are falsy so behavior is correct, but the type error degrades type safety and masks any real issues in the capture pipeline.
- **Confidence**: HIGH
- **Recommendation**: Either change `CaptureViewProps.newestCapture` to `ReceiverCapture | undefined`, or coerce with `newestCapture ?? null` at the call site.

## Medium Findings

### M1. Missing `JSX` namespace in `icons.tsx` [NEW]
- **File**: `packages/app/src/views/Receiver/icons.tsx:81`
- **Issue**: The type annotation `({ active }: ReceiverAppBarIconProps) => JSX.Element` references the global `JSX` namespace, which requires `@types/react` to be properly resolved. TS2503: "Cannot find namespace 'JSX'".
- **Impact**: The `receiverNavItems` array type is degraded. Since the icons are consumed via the `Icon` property in `ReceiverShell`, this could allow passing a non-component function without a type error.
- **Confidence**: HIGH
- **Recommendation**: Replace `JSX.Element` with `React.ReactElement` or `React.JSX.Element`, or ensure `@types/react` is installed and resolvable.

### M2. Duplicated `sizeLabel` utility function [NEW]
- **Files**: `packages/app/src/views/Receiver/CaptureView.tsx:25`, `packages/app/src/views/Receiver/InboxView.tsx:18`
- **Issue**: Identical `sizeLabel(byteSize: number)` function is defined in both files with the same logic (B/KB/MB formatting).
- **Impact**: Any fix or improvement needs to be applied in two places. Low risk but a clear DRY violation.
- **Confidence**: HIGH
- **Recommendation**: Extract to a shared utility in `packages/app/src/views/Receiver/` or a component helper.

### M3. 75 test type errors from missing `@testing-library/jest-dom` type augmentation [NEW]
- **Files**: All 5 test files under `packages/app/src/__tests__/`
- **Issue**: Tests use matchers like `.toBeVisible()`, `.toBeInTheDocument()`, `.toHaveAttribute()`, `.toHaveValue()`, `.toBeDisabled()` which come from `@testing-library/jest-dom`. These work at runtime through Vitest's setup, but TypeScript cannot see the type augmentation on `Assertion<HTMLElement>`.
- **Impact**: 75 TS2339 errors across test files. `tsc --noEmit` cannot be used as a CI gate for the app package without `skipLibCheck` or proper type setup. The tests themselves all pass.
- **Confidence**: HIGH
- **Recommendation**: Add `@testing-library/jest-dom` types to the test tsconfig, or add a `vitest-setup.d.ts` with the appropriate type reference.

### M4. `ReceiverApp.test.tsx` uses `exact` option not in `ByRoleOptions` [NEW]
- **File**: `packages/app/src/__tests__/ReceiverApp.test.tsx:44-46`
- **Issue**: Three calls to `screen.getByRole('heading', { exact: false, name: ... })` produce TS2769 errors because `exact` is not in `ByRoleOptions` type definition. This may be a version mismatch between `@testing-library/react` and its type definitions.
- **Impact**: Tests pass at runtime (the library accepts the option), but the type error indicates a potential API drift between the installed version and its types.
- **Confidence**: HIGH
- **Recommendation**: Verify `@testing-library/react` version matches its installed types. If the types lag behind, use `name: /partial match/i` regex instead of `exact: false`.
