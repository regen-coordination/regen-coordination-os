## Critical Findings

### C1. `privilegedActionTypeSchema` missing ERC-8004 and FVM action types [NEW]
- **File**: `packages/shared/src/contracts/schema.ts:89-96`
- **Issue**: The `privilegedActionTypeSchema` Zod enum defines only 6 values (`anchor-mode-toggle`, `archive-upload`, `archive-follow-up-refresh`, `archive-anchor`, `safe-deployment`, `green-goods-transaction`). However, the extension uses three additional action types that are not in this enum:
  - `'erc8004-registration'` тАФ used at `packages/extension/src/background/handlers/action-executors.ts:858,871`
  - `'erc8004-feedback'` тАФ used at `action-executors.ts:929,942`
  - `'fvm-register-archive'` тАФ used at `packages/extension/src/background/handlers/archive.ts:905,922,960,985`
- **Impact**: These are type errors at compile time (TS2322). At runtime, Zod validation of privileged action logs will reject these entries, potentially causing silent failures in the operator action log. Since these actions involve **onchain transactions** (ERC-8004 agent registration, FVM Filecoin registry), losing their audit trail is a security and compliance concern.
- **Recommendation**: Add `'erc8004-registration'`, `'erc8004-feedback'`, and `'fvm-register-archive'` to `privilegedActionTypeSchema` in `schema.ts`.

## High Findings

### H1. Missing `vite/client` types cause 40+ `import.meta.env` errors across extension [NEW]
- **File**: `packages/extension/tsconfig.json`, `tsconfig.base.json:20`
- **Issue**: The base tsconfig specifies `"types": ["chrome", "node"]` but does not include `"vite/client"`. No `vite-env.d.ts` file exists in any package. This causes `import.meta.env` to be unrecognized by tsc, producing TS2339 errors in:
  - `context.ts` (18 occurrences, lines 147-186)
  - `coop.ts` (2 occurrences, lines 96, 133)
  - `receiver-sync-offscreen.ts` (3 occurrences, lines 106-108)
  - `useDashboard.ts` (7 occurrences, lines 48-57)
  - `useSyncBindings.ts` (3 occurrences, lines 57-59)
- **Impact**: Type checking is degraded across the entire config resolution layer. The build works (Vite injects types at build time) but `tsc --noEmit` reports ~33 false errors, masking real type issues.
- **Recommendation**: Either add `"vite/client"` to `types` in `tsconfig.base.json`, or create a `vite-env.d.ts` (`/// <reference types="vite/client" />`) in each package that uses `import.meta.env`.

### H2. Missing `@types/react` causes ~2500 JSX type errors across app and extension [NEW]
- **File**: `packages/app/tsconfig.json`, `packages/extension/tsconfig.json`
- **Issue**: React 19 is installed but `@types/react` is either missing or not properly resolved. Every JSX element triggers TS7026 ("JSX element implicitly has type 'any' because no interface 'JSX.IntrinsicElements' exists"). This affects all `.tsx` files: `ReceiverShell.tsx`, `tabs.tsx`, `TabStrip.tsx`, `cards.tsx`, `SidepanelApp.tsx`, `ErrorBoundary.tsx`, all test files, etc.
- **Impact**: No type safety on any JSX props or elements. This completely defeats the purpose of TypeScript for the entire view layer. Compile-time prop mismatches go undetected.
- **Recommendation**: Ensure `@types/react` and `@types/react-dom` are installed and resolvable. React 19 may require `@types/react@19`. Verify `tsconfig` `jsx` setting and React JSX transform compatibility.

### H3. `setCoopArchiveSecrets` called with wrong argument shape [NEW]
- **File**: `packages/extension/src/background/handlers/archive.ts:627`
- **Issue**: The function `setCoopArchiveSecrets(db, coopId, secrets)` expects its third argument to be `CoopArchiveSecrets` which requires `coopId: string` and `proofs: string[]`. At line 627, `payload.secrets` is passed directly, but its inferred type is `{ agentPrivateKey?: string; spaceDelegation: string; proofs?: string[] }` тАФ missing the required `coopId` and with `proofs` as optional rather than required.
- **Impact**: TS2345 compile error. At runtime, Zod's `.parse()` inside `setCoopArchiveSecrets` may throw if the shape doesn't match, potentially crashing the archive config save flow.
- **Recommendation**: Spread `coopId` into the secrets argument: `await setCoopArchiveSecrets(db, payload.coopId, { ...payload.secrets, coopId: payload.coopId })`.

### H4. `PrivilegedFeature` type missing `'Filecoin registry registration'` string [NEW]
- **File**: `packages/extension/src/background/handlers/archive.ts:883`, `packages/extension/src/runtime/operator.ts:9-14`
- **Issue**: The `PrivilegedFeature` type union defines 5 string literals. At `archive.ts:883`, `requireAnchorModeForFeature` is called with `feature: 'Filecoin registry registration'`, which is not in the union (the closest match is `'erc8004 identity registration'`).
- **Impact**: TS2322 compile error. The error message shown to users when anchor mode is missing for FVM registration will be incorrect or the call may be rejected by type narrowing in strict mode.
- **Recommendation**: Add `'Filecoin registry registration'` to the `PrivilegedFeature` type in `operator.ts`, or rename the call site to match an existing value.

### H5. `ReviewDraft` schema drift тАФ test fixtures missing required fields [NEW]
- **File**: `packages/extension/src/background/handlers/__tests__/review-handlers.test.ts:126,142`
- **Issue**: Test fixture for `ReviewDraft` is missing `sourceCandidateId`, `extractId`, and `interpretationId` fields that are now required by the schema. The test uses `satisfies ReviewDraft` which catches this at compile time (TS1360/TS2739).
- **Impact**: Tests compile with errors. If these tests are skipped or run without type checking, they may pass but with invalid test data that doesn't match production shapes.
- **Recommendation**: Update test fixtures to include the three missing required fields.

### H6. `onchainState` initialized as `{}` instead of proper shape [NEW]
- **Files**: `packages/extension/src/views/shared/useCoopActions.ts:96`, `packages/extension/src/views/Sidepanel/hooks/useCoopForm.ts:100`
- **Issue**: Both files initialize `onchainState` as an empty object `{}`, but the type requires `chainId`, `chainKey`, `safeAddress`, `statusNote`, `safeCapability` as mandatory fields.
- **Impact**: TS2739 compile error. At runtime, any code accessing `onchainState.chainKey` or `.safeCapability` will get `undefined`, which could cause downstream failures in Safe deployment flows or chain resolution.
- **Recommendation**: Initialize with proper defaults: `{ chainId: 0, chainKey: 'sepolia', safeAddress: '', statusNote: '', safeCapability: 'unavailable' }`.

### H7. `agent-runner.ts` accesses non-existent output properties [NEW]
- **File**: `packages/extension/src/runtime/agent-runner.ts:1157,1161,1172`
- **Issue**: Three property access errors:
  - Line 1157: `typed.digestMarkdown` does not exist on `ReviewDigestOutput` type
  - Line 1161: Same property accessed again
  - Line 1172: `typed.rationale` does not exist on `CapitalFormationBriefOutput` type
- **Impact**: At runtime, these will be `undefined` rather than throwing, so the agent will silently produce empty memory entries for review-digest and capital-formation-brief skill outputs.
- **Recommendation**: Either add these fields to the output types in the schema, or use the correct property names from the actual output interfaces.

## Medium Findings

### M1. ZK proof generation hardcoded to `null` in review publish [NEW]
- **File**: `packages/extension/src/background/handlers/review.ts:58-61`
- **Issue**: Anonymous publish sets `membershipProof = null` with a TODO comment explaining that snarkjs WASM workers are unavailable in MV3 service workers. The offscreen document delegation mentioned in the TODO is not implemented.
- **Impact**: Anonymous publishing works but produces artifacts with no membership proof, meaning the anonymity claim is unverifiable by other coop members. This is a known architectural limitation documented by the TODO.

### M2. `BigInt` (constructor) used instead of `bigint` (primitive) in session handler [NEW]
- **File**: `packages/extension/src/background/handlers/session.ts:224,238,277`
- **Issue**: `BigInt(...)` constructor calls return a `BigInt` wrapper object, but the expected type is `bigint` (primitive). TypeScript distinguishes these (TS2322).
- **Impact**: In practice, `BigInt(x)` returns a primitive when not called with `new`, so this works at runtime. But it's a type-safety gap that could mask real issues.
- **Recommendation**: Cast or adjust type expectations. If using `BigInt()` as a conversion function, the TypeScript target may need `ES2020+` or explicit type annotations.

### M3. `inference-worker.ts` has nullable pipeline and push-on-wrong-type issues [NEW]
- **File**: `packages/extension/src/runtime/inference-worker.ts:65,83,111`
- **Issue**: Three type errors:
  - Line 65: `TextGenerationPipeline` assigned to incompatible function type
  - Line 83: `.push()` called on `{ criteria: unknown[] }` type (wrong shape)
  - Line 111: Pipeline is possibly null and invoked without null check
- **Impact**: If the pipeline fails to initialize or the criteria shape changes, the inference worker will crash without a descriptive error.

### M4. `agent-models.ts` accesses possibly-undefined ONNX WASM config [NEW]
- **File**: `packages/extension/src/runtime/agent-models.ts:163,165`
- **Issue**: `env.backends.onnx.wasm` is flagged as possibly undefined (TS18048), and the expression produces a union type too complex to represent (TS2590).
- **Impact**: If the ONNX backend is not available or the Transformers.js version changes its config shape, this will throw at runtime during model initialization.

### M5. `tabs.tsx` is a 1725-line god object with multiple type mismatches [NEW]
- **File**: `packages/extension/src/views/Sidepanel/tabs.tsx`
- **Issue**: This single file contains the entire sidepanel tab rendering logic. It has 8+ type errors where component props don't match their declared types (`DraftCardProps`, `ArtifactCardProps`, `ArchiveReceiptCardProps`, `ReceiverIntakeCardProps`, `OperatorConsoleProps`). Also accesses non-existent `canBadgeApp` on `BrowserUxCapabilities` (line 1577).
- **Impact**: Every schema change that adds a field to the draft/artifact/receipt types requires updating this file. The type mismatches indicate the components and their callers have drifted apart.
- **Recommendation**: Split into per-tab modules. Fix prop interfaces to match actual data shapes.

### M6. `sw-safety.test.ts` AST walker uses incorrect type comparisons [NEW]
- **File**: `packages/extension/src/__tests__/sw-safety.test.ts:165-209`
- **Issue**: 9 type comparison errors (TS2367) where `node.type` (narrowed to `"Program"`) is compared against `"ImportExpression"`, `"MemberExpression"`, `"Identifier"`, `"CallExpression"`, `"NewExpression"`. The `walkTopLevel` function appears to narrow the node type too aggressively.
- **Impact**: Test compiles with errors but likely works at runtime because the comparisons still execute. The issue is that TypeScript's control flow analysis thinks these comparisons can never be true, which means the type definitions for the AST walker may be incorrect.

### M7. 24 lint errors including formatting and import ordering [NEW]
- **Files**: Multiple files across packages (see `bun lint` output)
- **Issue**: Biome reports 24 errors: import sorting issues in docs components, formatting issues in `tab-router/skill.ts` and `useTabCapture.ts`, and `noArrayIndexKey` lint rule violation in `Navbar/Content/index.tsx`.
- **Impact**: CI would fail if lint is a gate. These are auto-fixable with `bun lint --fix`.

### M8. `receiver.ts:258` тАФ receiver capture shape mismatch [NEW]
- **File**: `packages/extension/src/background/handlers/receiver.ts:258`
- **Issue**: The `capture` object passed to `saveReceiverCapture` has `syncState: string` (literal `'synced'`), but the expected type requires `syncState` to be one of `"failed" | "local-only" | "queued" | "synced"`. Also, `kind` field is `"link" | "file" | "audio" | "photo"` but the spread object includes `pairingId`, `coopId`, etc. that are not part of the `ReceiverCapture` shape for `saveReceiverCapture`.
- **Impact**: Type error at compile time. At runtime, extra properties are harmless, but the type mismatch suggests the `saveReceiverCapture` function's expected type may have changed without updating the caller.
